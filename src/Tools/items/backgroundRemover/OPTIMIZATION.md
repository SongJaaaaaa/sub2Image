# 抠图画质问题与优化方向

## 1. 当前已知问题

### 边缘细节不够精细

常见表现：

- 毛发、绒毛和细线结构被截断。
- 半透明区域变成硬边。
- 原背景颜色残留在主体边缘，形成白边或色边。

主要原因：

- `background-removal-js` 的 IS-Net 固定使用 `1024x1024` 输入，原图中的细小结构可能在模型推理前已经丢失。
- 低分辨率、压缩和运动模糊造成的细节无法通过后处理重新生成。
- 当前边缘范围、柔化和去色边只处理已有 Alpha，不能恢复模型完全没有识别到的发丝。

### 主体部分被误删

常见表现：

- 衣服、手指、饰品或与背景颜色接近的区域被当成背景。
- 主体内部出现透明空洞。
- 多个物体时，模型只保留其中一个显著主体。

这是主体分割错误，不是边缘透明度问题。继续扩大或柔化 Alpha 只能改善边缘，不能可靠恢复已经被模型删除的像素。

## 2. 当前浏览器方案的能力边界

当前实现：

- 高清细节：完整精度 `isnet`。
- 标准快速：半精度 `isnet_fp16`。
- 后处理：边缘范围、边缘柔化、去除色边。
- 预设：原始边缘、毛发细节、商品净边。

适合：

- 主体轮廓清晰、背景反差明显的单主体图片。
- 不允许上传服务器或需要离线处理的场景。

不适合仅靠后处理解决：

- 模型完全漏掉的发丝或主体区域。
- 主体和背景高度相似。
- 玻璃、薄纱、烟雾等复杂透明材质。
- 需要用户指定保留对象的多主体图片。

## 3. 服务端模型实验

测试服务位于项目根目录 `matting-server/`，使用官方 `ZhengPeng7/BiRefNet-matting`：

- 模型提交固定为 `57f9f68b43ba337c75762b14cf3075d659007268`。
- CPU 固定 4 线程、单进程、单并发。
- 图片在内存中处理，不主动保存原图和结果。
- 输入最长边限制为 `2048px`。

4 核、4GB 内存、无 GPU 的临时服务器实测：

| 输入 | 耗时 | 结果 |
| --- | ---: | --- |
| 1200x675 测试图 | 41.82 秒 | 成功输出 RGBA PNG |
| 692x691 人像测试图 | 41.92 秒 | 脸、眼镜、耳朵、衣领完整，Alpha 无内部空洞 |

当前结论：

- `BiRefNet-matting` 对主体完整性和人像边缘明显优于当前浏览器 IS-Net，适合作为服务端高清模式的第一候选。
- 人像测试图本身分辨率较低，头发已经与原背景混色，输出仍会有轻微灰边。
- 暂时没有证据表明必须再串联 ViTMatte，应先扩大问题样本测试。

## 4. 后续模型选择

### BiRefNet-matting

优先级最高。它是无 Trimap 的 Matting 模型，一次推理同时处理主体和半透明边缘，接入成本最低。

### ViTMatte

只在 BiRefNet 主体判断正确但毛发、薄纱等 Alpha 仍不够精细时增加。ViTMatte 需要 Trimap，可从 BiRefNet 遮罩自动生成：

```text
BiRefNet 遮罩
  -> 腐蚀得到确定前景
  -> 膨胀得到未知边缘
  -> ViTMatte 只精修未知边缘
```

ViTMatte 不负责决定主体是谁，不能单独解决主体误删。

### SAM 2 或框选主体

只在多主体、主体不显著或自动模型经常选错对象时考虑。SAM 更适合根据点击或框选确定对象，不负责毛发 Alpha，仍需配合 Matting。

### 局部恢复遮罩

作为所有自动模型失败时的补救。允许用户在原图上恢复或擦除局部区域，但不把手工描边作为默认流程。

## 5. 推荐实现顺序

1. 收集至少 20 张失败样图，覆盖毛发、透明物、同色背景、内部空洞、多主体和贴边主体。
2. 对同一批图片保存 IS-Net、BiRefNet-matting 的透明 PNG 和 Alpha 遮罩。
3. 在网格、白底、黑底三种背景下检查边缘，不只看单一预览背景。
4. 确认 BiRefNet-matting 的误删率和边缘质量后，再接入前端“服务端高清”模式。
5. 只有 BiRefNet Alpha 仍明显不足时，增加 ViTMatte 对照组。
6. 最后增加局部恢复遮罩，处理模型不可避免的少量失败图片。

## 6. 部署建议

正式产品保留两种模式：

- 本地快速：继续使用 `background-removal-js`，图片不上传。
- 服务端高清：上传到自有服务器运行 `BiRefNet-matting`，返回透明 PNG。

正式服务器推荐 NVIDIA GPU 和至少 12GB 显存。4 核、4GB CPU 服务器只适合单张效果验证，不适合并发生产。

上线前必须分别核对代码仓库和模型权重的实际许可证，不要只根据 GitHub 仓库许可证判断模型权重可商用。

临时服务器验证结束后，建议删除远端测试容器、模型缓存和临时 Swap，只保留仓库中的 `matting-server/`。服务器密码曾用于临时测试时，还应立即轮换密码。
