# Tools 与 Skills 拓展走向

## 1. 文档定位

本文档定义项目后续 Tools、Skills 和独立拓展工作区的目标结构、模块边界、开发顺序及配套文档规范。

本文档是这部分功能的权威设计依据。`docs/project-structure-and-store-refactor-plan.md` 中关于 `src/extensions/`、Skill、Workflow、Tool 和扩展侧边栏的旧规划，以本文档为准；其他 Store、Feature 和组件拆分规划不受影响。

当前仓库已经存在尚未完成的 `src/extensions/` 和 `src/components/workspaceSidebar/`。正式开发拓展工作区前，应先审查并迁移其中仍有价值的代码，不再继续向旧目录添加实现，也不要让新旧两套注册表长期并存。

## 2. 已确认的产品方向

1. 原有画廊、Agent、生成输入区和已有工作流保持不变。
2. 原应用只新增一个顶层入口，用于打开独立拓展工作区。
3. 拓展工作区拥有自己的侧边栏，目前只分为 `Tools` 和 `Skills`。
4. `Tools` 是用户直接操作的独立功能应用。
5. `Skills` 是指导文本模型生成图片描述或提示词的指令包。
6. Tools 和 Skills 可以复用原项目的图片库、图片 API、文本模型和持久化能力，但不能直接嵌入或改造原有业务页面。
7. 第一个 Tool 是基于 Filerobot Image Editor 的图片编辑器。
8. 第一阶段不引入 Fabric.js、视频编辑、公开市场和 Workflow 分类。
9. Tool 编辑产生的新图片保存为新资源，不覆盖原图。
10. 用户导入的 Skill 只能包含声明和文本指令，不执行任意 JavaScript、HTML 或系统命令。

## 3. 概念边界

### 3.1 Workspace Tool

Workspace Tool 是拓展工作区中的独立应用，有自己的页面、状态和交互流程。

示例：

- 图片编辑器
- 多图拼接器
- 背景移除器
- 视频处理器

Tool 可以调用底层图片或视频能力，但不负责指导对话模型生成提示词。

### 3.2 Generation Skill

Generation Skill 是生成图片描述或提示词时使用的指令包。

示例：

- 电商产品图
- 角色设定图
- 分镜描述
- 建筑效果图

Skill 可以包含提示词规则、默认参数、问题清单和示例，但不提供图片编辑器 UI，也不直接修改图片像素。

### 3.3 Agent Function Tool

Agent Function Tool 是模型通过 Responses API 调用的函数，例如 `generate_image`、`generate_image_batch`。它属于模型执行协议，不是侧边栏中的 Workspace Tool。

代码和文档必须使用下面的完整名称，避免继续把三个概念都简称为 Tool：

| 用户界面概念 | 代码命名 |
| --- | --- |
| 侧边栏工具应用 | `WorkspaceTool` |
| 图片描述技能 | `GenerationSkill` |
| 模型函数工具 | `AgentFunctionTool` |

## 4. 页面与入口关系

```text
原有应用
├── 画廊
├── Agent
├── 原有图片生成流程
└── 新入口：拓展工作区
              │
              ├── Tools
              │   ├── 图片编辑器
              │   └── 后续其他工具
              │
              └── Skills
                  ├── 内置技能
                  ├── 用户导入技能
                  └── 技能运行页
```

推荐路由：

```text
/app                              原有应用
/app/extensions                   拓展工作区默认页

/app/extensions/tools/:toolId     Tool 运行页

/app/extensions/skills/:skillId   Skill 详情或运行页
```

第一阶段不要求引入新的路由库。可以沿用项目当前的 `window.history` 和 pathname 状态，但路径解析必须集中在拓展工作区入口，不能散落在 Tool 或 Skill 子模块中。

## 6. 根模块职责

### 6.1 `src/ExtensionWorkspace`

只负责拓展工作区外壳：

- 拓展工作区路由解析。
- Tools、Skills 两个一级入口。
- 独立侧边栏、头部和内容区域布局。
- 返回原应用。
- 加载 Tool 或 Skill 的公共入口。

它不负责：

- 图片编辑实现。
- Skill 指令解析。
- 图片或提示词持久化。
- 直接调用图片 API。

### 6.2 `src/Tools`

只负责 Workspace Tool：

- Tool 类型和注册表。
- Tool 列表与宿主容器。
- Tool 懒加载。
- Tool 与图片库之间的适配层。
- 每个具体 Tool 的实现。

`src/Tools` 不得导入 `src/Skills` 内部文件，也不得把 Filerobot 逻辑放入全局 Store。

### 6.3 `src/Skills`

只负责 Generation Skill：

- 内置 Skill 注册。
- 用户 Skill 导入、导出和本地存储。
- Skill 指令读取和运行。
- 调用文本模型生成图片描述。
- 在拓展工作区内继续调用图片生成能力。

`src/Skills` 不得依赖具体 Tool，不得通过 Filerobot 处理图片。

## 7. 依赖方向

```text
原应用入口
  -> ExtensionWorkspace
      -> Tools 公共出口
      -> Skills 公共出口

Tools
  -> Tools/adapters
      -> 图片库公共 API
      -> 图片存储公共 API

Skills
  -> Skills/adapters
      -> 文本模型公共 API
      -> 图片生成公共 API
      -> 图片库公共 API
```

强制约束：

1. `ExtensionWorkspace` 只能从 `Tools/index.ts`、`Skills/index.ts` 导入。
2. `Tools` 和 `Skills` 不能相互导入内部文件。
3. 具体 Tool 只能通过 `Tools/adapters` 访问原项目能力。
4. 具体 Skill 只能通过 `Skills/adapters` 访问原项目能力。
5. 不从 `src/store.ts` 深入读取或修改不属于拓展模块的状态。
6. 不导入原画廊、Agent 或 Composer 的页面组件。
7. 原业务页面除了“打开拓展工作区”的入口外，不反向依赖具体 Tool 或 Skill。
8. 跨模块能力只能从各模块 `index.ts` 公共出口导入。

数据层复用不等于 UI 交叉。Tools 和 Skills 可以读取同一图片库、使用同一 API 配置，但必须拥有独立页面和操作流程。

## 8. Workspace Tool 契约

第一版保持简单，不建立复杂插件框架。

```ts
export type WorkspaceTool = {
  id: string
  name: string
  description: string
  version: number
  icon?: string
  load: () => Promise<{ default: React.ComponentType }>
}
```

注册规则：

1. `id` 全局唯一，使用小写短横线形式，例如 `image-editor`。
2. 每个 Tool 在自己的 `definition.ts` 定义元数据。
3. `registry.ts` 只组合 definition，不包含 Tool 业务逻辑。
4. Tool 页面通过动态 `import()` 懒加载，避免 Filerobot 进入原应用首屏包。
5. Tool 不允许修改其他 Tool 的内部状态。
6. 每个 Tool 根目录必须有 `README.md`。

新增 Tool 时，只修改以下位置：

- 新增 `src/Tools/items/<toolName>/`。
- 在 `src/Tools/registry.ts` 注册 definition。
- 必要时扩充 `Tools/adapters` 的公共能力。
- 更新 `src/Tools/README.md` 中的已注册 Tool 列表。

## 9. 第一个 Tool：Filerobot 图片编辑器

### 9.1 用户流程

```text
打开拓展工作区
  -> Tools
  -> 图片编辑器
  -> 选择图片来源
      -> 上传本地图片
      -> 从现有图片库选择
  -> 打开 Filerobot
  -> 滤镜、裁剪、旋转、文字、水印等编辑
  -> 保存
  -> 作为新图片写入图片库
```

### 9.2 实现边界

- Filerobot 仅存在于 `src/Tools/items/imageEditor/`。
- Filerobot 的 tabs、tools、主题和中文文案集中在 `lib/filerobotConfig.ts`。
- 图片来源选择集中在 `ImageSourcePicker.tsx`。
- 图片读取通过 `Tools/adapters/imageLibrary.ts`。
- 图片保存通过 `Tools/adapters/imageStorage.ts`。
- 保存结果必须创建新图片，保留来源图片。
- 关闭编辑器时未保存的修改不写入 IndexedDB。
- 第一阶段只处理单张图片。
- 第一阶段不实现复杂多图图层、拼图和视频编辑。

### 9.3 技术要求

- 使用 `react-filerobot-image-editor`，按需安装兼容的 `react-konva` 和 `styled-components`。
- 使用动态 import 加载编辑器组件。
- 导出格式至少支持 PNG 和 JPEG。
- 中文字体在导出前必须确认加载完成，避免文字缺字或乱码。
- 图片加载或保存失败时记录 `console.warn` 或 `console.error`，并显示中文 Toast。
- 不为没有证据的失败路径增加大量兼容分支；遇到无法复现的问题先补日志。

## 10. Generation Skill 契约

Skill 的核心作用是把用户的简短需求转为更完整的图片描述或提示词。

```ts
export type GenerationSkill = {
  id: string
  name: string
  description: string
  version: number
  instructions: string
  supports: Array<'text-to-image' | 'image-to-image'>
  defaults?: {
    size?: string
    quality?: string
    count?: number
  }
}
```

Skill 可以做：

- 规定描述结构。
- 补充构图、镜头、材质、光线和风格信息。
- 根据参考图生成编辑描述。
- 提出必要问题。
- 提供图片尺寸、质量和数量默认值。
- 输出单张或多张图片描述。

Skill 不可以做：

- 执行 JavaScript、HTML、Shell 或系统命令。
- 读取 API Key、浏览器存储或其他 Skill 数据。
- 直接访问网络。
- 修改原项目系统指令和 Agent Function Tool 策略。
- 直接处理图片像素。

## 11. 用户 Skill 包格式

推荐使用 zip 包或目录导入，最小结构如下：

```text
product-photo.skill/
├── skill.json
├── SKILL.md
├── cover.webp       可选
└── examples/        可选
```

`skill.json` 保存可机器读取的元数据：

```json
{
  "id": "product-photo",
  "name": "电商产品图",
  "description": "生成专业产品摄影描述",
  "version": 1,
  "supports": ["text-to-image", "image-to-image"],
  "defaults": {
    "size": "1024x1024",
    "quality": "high"
  }
}
```

`SKILL.md` 保存文本模型需要读取的说明。导入时只读取允许的文本和图片资源，忽略其他文件。

外部 Skill 属于不可信输入，必须校验：

- `skill.json` 是否存在且可以解析。
- 必填字段和版本是否有效。
- `id` 是否与现有 Skill 冲突。
- `SKILL.md` 是否存在且在合理大小范围内。
- 图片资源格式是否允许。
- 压缩包内路径不得逃逸目标目录。

这类校验属于外部输入边界，不违反项目的“简单优先”原则。

## 12. Skill 用户流程

### 12.1 管理

```text
打开拓展工作区
  -> Skills
  -> 查看内置和已导入 Skill
  -> 导入、查看、编辑、复制、导出或删除
```

### 12.2 使用

```text
选择一个 Skill
  -> 输入简短图片需求
  -> 可选参考图片
  -> 文本模型读取 Skill 指令
  -> 生成完整图片描述
  -> 用户确认或继续生成图片
  -> 结果写入共享图片库
```

第一阶段 Skill 的选择和运行完全位于拓展工作区，不向原有对话输入框增加选择器。

## 13. 模块文档规范

### 13.1 根模块必备文档

`src/ExtensionWorkspace`、`src/Tools`、`src/Skills` 根目录必须包含：

- `README.md`：模块用途、非目标、入口、当前功能和快速定位。
- `AGENTS.md`：约束 AI 在该目录中的开发方式和必读文档。
- `docs/`：架构、契约、接入步骤、边界和测试说明。

缺少上述文档时，不得开始在该根模块中堆叠新功能。

### 13.2 每个子模块必备说明

每个具体 Tool 和内置 Skill 根目录必须有 `README.md`。README 至少包含：

1. 模块作用。
2. 用户入口和完整流程。
3. 目录内主要文件职责。
4. 输入和输出。
5. 依赖的公共适配器。
6. 持久化数据及兼容要求。
7. 当前限制和明确非目标。
8. 测试位置和运行方式。
9. 新增或修改功能时必须同步更新的文档。

### 13.3 文档命名

- 根说明固定使用 `README.md`。
- AI 目录指令固定使用 `AGENTS.md`。
- 详细文档放入模块自己的 `docs/`。
- 文档文件名使用小写短横线，例如 `adding-a-tool.md`。
- 代码注释和项目文档默认使用中文。

## 15. 开发阶段

### 阶段 0：收敛旧扩展代码

- 审查当前未完成的 `src/extensions/` 和 `src/components/workspaceSidebar/`。
- 将可复用的注册表、测试或状态逻辑迁移到新结构。
- 删除已经迁移完成的旧入口，避免双注册表。
- 暂不保留 Workflow 一级分类。
- 迁移只改变目录和边界，不同时新增 Filerobot 业务。

### 阶段 1：独立拓展工作区

- 新增原应用顶层入口。
- 建立 `/app/extensions` 路径和独立页面外壳。
- 侧边栏只包含 Tools、Skills。
- 完成桌面和移动端基本导航。
- 原画廊和 Agent 页面行为保持不变。

### 阶段 2：Filerobot 图片编辑 Tool

- 建立 Tool 类型、注册表和懒加载宿主。
- 实现本地上传和现有图片库选择。
- 接入 Filerobot。
- 支持滤镜、裁剪、旋转、文字、水印和导出。
- 保存结果为新图片。
- 补齐 Tool 文档和测试。

### 阶段 3：Skills 管理

- 建立 Skill 类型和注册表。
- 展示内置 Skill。
- 实现 Skill 详情、启用和运行入口。
- 完成至少一个内置 Skill。
- 暂不实现外部 Skill 上传。

### 阶段 4：Skill 运行和图片生成

- 接入文本模型适配器。
- 根据 Skill 指令生成图片描述。
- 支持用户确认描述后调用图片生成。
- 生成结果写入共享图片库。

### 阶段 5：Skill 导入与导出

- 实现 `skill.json`、`SKILL.md` 和可选资源解析。
- 添加外部输入校验。
- 存储用户导入 Skill。
- 支持复制、编辑、导出和删除。

### 阶段 6：后续能力

按真实需求逐项评估：

- AI 创建或修改 Skill。
- 更多独立 Tool。
- 多图拼接和自由画布。
- 背景移除。
- 视频处理 Tool。
- Tool 或 Skill 分享与 Remix。

这些能力不得提前污染第一阶段接口。

## 16. 测试策略

### 16.1 单元测试

- Tool 和 Skill 注册表 ID 去重。
- 路由解析。
- Skill 元数据和导入格式校验。
- 图片来源转换和保存。
- Filerobot 输出处理。
- Skill 指令组装。

### 16.2 边界测试

使用源码扫描测试保证：

- `ExtensionWorkspace` 不导入 Tool 或 Skill 内部文件。
- `Tools/items` 不直接导入原业务页面和 Store 内部实现。
- `Skills/builtins` 不直接导入文本模型和图片生成内部实现。
- `Tools` 与 `Skills` 不互相导入内部文件。
- 原业务页面不导入具体 Tool 或 Skill。
- 不再从旧 `src/extensions` 注册新模块。

### 16.3 UI 测试

- 图片编辑器选择现有图片。
- 保存编辑结果且原图仍存在。
- Skill 列表、详情和运行。


## 17. 每个模块的完成标准

一个 Tool 或 Skill 只有同时满足以下条件才算完成：

1. 拥有自己的根目录和 `README.md`。
2. 通过公共注册表接入。
3. 不绕过 adapters 访问原业务内部实现。
4. 用户流程完整，没有 stub、TODO 或不可用按钮。
5. 外部输入有与风险匹配的必要校验。
6. 失败路径有中文提示和必要日志。
7. 有对应单元测试或 UI 测试。
8. `npm run build` 和相关测试通过。
9. 行为、限制或目录变化已经同步到文档。

## 18. 当前非目标

- 不重做原有画廊和 Agent。
- 不在原对话输入区加入 Skill 选择器。
- 不让 Filerobot 成为全局图片处理引擎。
- 不开发通用 Fabric.js 编辑器。
- 不执行用户上传代码。
- 不建立 Tool/Skill 在线市场。
- 不开发视频剪辑。
- 不恢复 Workflow 一级入口。
- 不为未来能力提前引入复杂插件框架、事件总线或多 Store 同步。

## 19. 命名与迁移注意事项

- 目录名固定为 `src/Tools`、`src/Skills`、`src/ExtensionWorkspace`。
- React 组件使用 PascalCase 文件名。
- 普通 TypeScript 文件使用小写驼峰文件名。
- Tool、Skill ID 使用小写短横线。
- macOS 默认文件系统通常不区分大小写，迁移旧目录时不要依赖只修改大小写的普通重命名。
- 迁移当前未提交代码前先阅读并保留其有效实现，不使用破坏性 Git 命令。
- `src/extensions` 清理完成后，不保留指向新目录的长期兼容层；在同一迁移阶段修改调用方。

## 20. 最终原则

1. 原应用与拓展工作区 UI 独立。
2. Tools 与 Skills 概念独立。
3. 模块通过公共出口和 adapters 共享底层能力。
4. 每个根模块先有说明和规范，再增加功能。
5. 每个具体 Tool 或 Skill 都有自己的 README。
6. AI 开发前先读取对应模块文档，开发后同步更新文档。
7. 第一阶段优先交付完整可用的 Filerobot Tool，不提前建设通用插件平台。
