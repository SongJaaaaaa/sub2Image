# Tools 与 Skills 拓展走向

## 1. 文档定位

本文档定义项目后续 Tools、Skills 和独立拓展工作区的目标结构、模块边界、开发顺序及配套文档规范。

本文档是这部分功能的权威设计依据。`docs/project-structure-and-store-refactor-plan.md` 中关于 `src/extensions/`、Skill、Workflow、Tool 和扩展侧边栏的旧规划，以本文档为准；其他 Store、Feature 和组件拆分规划不受影响。

当前仓库已经存在尚未完成的 `src/extensions/` 和 `src/components/workspaceSidebar/`。正式开发拓展工作区前，应先审查并迁移其中仍有价值的代码，不再继续向旧目录添加实现，也不要让新旧两套注册表长期并存。

## 2. 已确认的产品方向

1. 原有画廊、Agent、生成输入区和已有工作流保持不变。
2. 原应用只新增一个顶层入口，用于打开独立拓展工作区。
3. 拓展工作区拥有自己的侧边栏，目前只分为 `Tools` 和 `Skills`。
4. `Tools` 是用户直接操作的独立功能应用。
5. `Skills` 是用户在 Agent 对话中手动选择的图像和视频创作指令包，第一批围绕生图、图生图、分镜与视频提示词。
6. Tools 可以复用原项目的图片库、图片 API 和持久化能力；Skills 通过公共出口接入 Agent 请求，但不能依赖具体 Tool。
7. 第一个 Tool 是基于 Filerobot Image Editor 的图片编辑器。
8. 第一阶段不引入 Fabric.js、视频编辑、公开市场和 Workflow 分类。
9. Tool 编辑产生的新图片保存为新资源，不覆盖原图。
10. 用户导入的 Skill 只能包含声明和文本指令，不执行任意 JavaScript、HTML 或系统命令。

## 3. 概念边界

### 3.1 Workspace Tool

Workspace Tool 是拓展工作区中的独立应用，有自己的页面、状态和交互流程。

示例：

- 图片编辑器
- 多图拼接器
- 背景移除器
- 视频处理器

Tool 可以调用底层图片或视频能力，但不负责指导对话模型生成提示词。

### 3.2 Agent Skill

Agent Skill 是用户在对话输入框中手动 `@` 的任务指令包。

示例：

- 电商产品图
- 角色一致性
- 图片编辑
- 海报与排版
- 分镜创作
- 视频生成提示词

Skill 可以包含工作流程、领域规则、问题清单和输出要求，但不提供独立工具 UI，也不直接执行代码。Skill 不自动检测或自动启用，只在用户明确选择的当前消息中生效。

### 3.3 Agent Function Tool

Agent Function Tool 是模型通过 Responses API 调用的函数，例如 `generate_image`、`generate_image_batch`。它属于模型执行协议，不是侧边栏中的 Workspace Tool。

代码和文档必须使用下面的完整名称，避免继续把三个概念都简称为 Tool：

| 用户界面概念 | 代码命名 |
| --- | --- |
| 侧边栏工具应用 | `WorkspaceTool` |
| Agent 指令技能 | `AgentSkill` |
| 模型函数工具 | `AgentFunctionTool` |

```text

              │

                  ├── 内置技能列表
```

推荐路由：


第一阶段不要求引入新的路由库。可以沿用项目当前的 `window.history` 和 pathname 状态，但路径解析必须集中在拓展工作区入口，不能散落在 Tool 或 Skill 子模块中。

## 6. 根模块职责

### 6.1 `src/ExtensionWorkspace`

只负责拓展工作区外壳：

- 拓展工作区路由解析。
- Tools、Skills 两个一级入口。
- 独立侧边栏、头部和内容区域布局。
- 返回原应用。
- 加载 Tool 或 Skill 的公共入口。

它不负责：

- 图片编辑实现。
- Skill 指令解析。
- 图片或提示词持久化。
- 直接调用图片 API。

### 6.2 `src/Tools`

只负责 Workspace Tool：

- Tool 类型和注册表。
- Tool 列表与宿主容器。
- Tool 懒加载。
- Tool 与图片库之间的适配层。
- 每个具体 Tool 的实现。

`src/Tools` 不得导入 `src/Skills` 内部文件，也不得把 Filerobot 逻辑放入全局 Store。

### 6.3 `src/Skills`

只负责 Agent Skill：

- 单文件 Markdown 解析和内置 Skill 注册。
- Skill 列表、详情和开源归属展示。
- 结构化 mention 的创建、识别和正文分离。
- 根据持久化的 ID 和版本解析请求指令。

`src/Skills` 不得依赖具体 Tool，不得通过 Filerobot 处理图片。

## 7. 依赖方向

```text
原应用入口
  -> ExtensionWorkspace
      -> Tools 公共出口
      -> Skills 公共出口

Tools
  -> Tools/adapters
      -> 图片库公共 API
      -> 图片存储公共 API

Skills
  -> Skills 公共出口
      -> Agent Composer
      -> Agent 请求组装
```

强制约束：

1. `ExtensionWorkspace` 只能从 `Tools/index.ts`、`Skills/index.ts` 导入。
2. `Tools` 和 `Skills` 不能相互导入内部文件。
3. 具体 Tool 只能通过 `Tools/adapters` 访问原项目能力。
4. Agent 和 Composer 只能从 `Skills/index.ts` 使用 Skill 公共能力。
5. 不从 `src/store.ts` 深入读取或修改不属于拓展模块的状态。
6. Skills 不导入原画廊、Agent 或 Composer 的页面组件。
7. 原业务代码不得直接导入具体 `builtins/*.md`。
8. 跨模块能力只能从各模块 `index.ts` 公共出口导入。

Tools 继续保持独立工具页面；Skills 的管理和说明位于拓展工作区，实际选择入口位于 Agent Composer。

## 8. Workspace Tool 契约

第一版保持简单，不建立复杂插件框架。

```ts
export type WorkspaceTool = {
  id: string
  name: string
  description: string
  version: number
  icon?: string
  load: () => Promise<{ default: React.ComponentType }>
}
```

注册规则：

1. `id` 全局唯一，使用小写短横线形式，例如 `image-editor`。
2. 每个 Tool 在自己的 `definition.ts` 定义元数据。
3. `registry.ts` 只组合 definition，不包含 Tool 业务逻辑。
4. Tool 页面通过动态 `import()` 懒加载，避免 Filerobot 进入原应用首屏包。
5. Tool 不允许修改其他 Tool 的内部状态。
6. 每个 Tool 根目录必须有 `README.md`。

新增 Tool 时，只修改以下位置：

- 新增 `src/Tools/items/<toolName>/`。
- 在 `src/Tools/registry.ts` 注册 definition。
- 必要时扩充 `Tools/adapters` 的公共能力。
- 更新 `src/Tools/README.md` 中的已注册 Tool 列表。

## 9. 第一个 Tool：Filerobot 图片编辑器

### 9.1 用户流程

```text
打开拓展工作区
  -> Tools
  -> 图片编辑器
  -> 选择图片来源
      -> 上传本地图片
      -> 从现有图片库选择
  -> 打开 Filerobot
  -> 滤镜、裁剪、旋转、文字、水印等编辑
  -> 保存
  -> 作为新图片写入图片库
```

### 9.2 实现边界

- Filerobot 仅存在于 `src/Tools/items/imageEditor/`。
- Filerobot 的 tabs、tools、主题和中文文案集中在 `lib/filerobotConfig.ts`。
- 图片来源选择集中在 `ImageSourcePicker.tsx`。
- 图片读取通过 `Tools/adapters/imageLibrary.ts`。
- 图片保存通过 `Tools/adapters/imageStorage.ts`。
- 保存结果必须创建新图片，保留来源图片。
- 关闭编辑器时未保存的修改不写入 IndexedDB。
- 第一阶段只处理单张图片。
- 第一阶段不实现复杂多图图层、拼图和视频编辑。

### 9.3 技术要求

- 使用 `react-filerobot-image-editor`，按需安装兼容的 `react-konva` 和 `styled-components`。
- 使用动态 import 加载编辑器组件。
- 导出格式至少支持 PNG 和 JPEG。
- 中文字体在导出前必须确认加载完成，避免文字缺字或乱码。
- 图片加载或保存失败时记录 `console.warn` 或 `console.error`，并显示中文 Toast。
- 不为没有证据的失败路径增加大量兼容分支；遇到无法复现的问题先补日志。

## 10. Agent Skill 契约

Skill 的核心作用是为用户手动选择的当前 Agent 消息提供专门工作方法。

```ts
export type AgentSkill = {
  id: string
  name: string
  description: string
  version: number
  author: string
  source: string
  license: string
  order: number
  instructions: string
}
```

Skill 可以做：

- 规定图像、图生图、分镜或视频提示词流程。
- 提出必要问题。
- 约束输出结构和质量检查。
- 在当前 Agent 工具能力范围内生成图片、关键帧或可复制提示词。

Skill 不可以做：

- 执行 JavaScript、HTML、Shell 或系统命令。
- 读取 API Key、浏览器存储或其他 Skill 数据。
- 假装已经读取文件、运行命令或验证结果。
- 修改原项目系统指令和 Agent Function Tool 策略。
- 自动启用自己或其他 Skill。
- 不虚构视频、音频或其他尚未接入的工具执行结果。

## 11. 内置 Skill 单文件格式

内置 Skill 平铺在 `src/Skills/builtins/`，一个 Skill 对应一个 Markdown：

```text
src/Skills/builtins/
├── product-photography.md
└── video-prompting.md
```

Markdown frontmatter 保存可机器读取的元数据：

```md
---
id: product-photography
name: 电商产品图
description: 为商品生成专业摄影提示词
version: 1
author: Anthropic
source: https://github.com/anthropics/skills
license: Apache-2.0
order: 10
---

Skill instructions...
```

`registry.ts` 自动读取所有 Markdown 并检查必填字段、ID 格式、版本和重复 ID。当前只加载仓库内可信文件，外部上传另行设计。

## 12. Skill 用户流程

### 12.1 查看

```text
打开拓展工作区
  -> Skills
  -> 查看平铺内置 Skill
  -> 查看职责、来源、许可证和完整指令
```

### 12.2 使用

```text
在画廊或 Agent 主输入框输入 @
  -> 从菜单手动选择一个 Skill
  -> 继续输入名称、ID 或职责关键词收紧筛选
  -> 输入本轮需求并发送
  -> 画廊输入框自动转到 Agent 对话
  -> 消息正文与 Skill 元数据分别持久化
  -> Agent 仅在本轮请求中读取 Skill 指令
```

每条消息最多选择一个 Skill。没有结构化 Skill mention 时，不注入任何 Skill 指令；普通 `@文字` 不触发 Skill。

## 13. 模块文档规范

### 13.1 根模块必备文档

`src/ExtensionWorkspace`、`src/Tools`、`src/Skills` 根目录必须包含：

- `README.md`：模块用途、非目标、入口、当前功能和快速定位。
- `AGENTS.md`：约束 AI 在该目录中的开发方式和必读文档。
- `docs/`：架构、契约、接入步骤、边界和测试说明。

缺少上述文档时，不得开始在该根模块中堆叠新功能。

### 13.2 每个子模块必备说明

每个具体 Tool 根目录必须有 `README.md`。内置 Skill 使用单文件正文和 frontmatter 自描述，不额外创建 README。Tool README 至少包含：

1. 模块作用。
2. 用户入口和完整流程。
3. 目录内主要文件职责。
4. 输入和输出。
5. 依赖的公共适配器。
6. 持久化数据及兼容要求。
7. 当前限制和明确非目标。
8. 测试位置和运行方式。
9. 新增或修改功能时必须同步更新的文档。

### 13.3 文档命名

- 根说明固定使用 `README.md`。
- AI 目录指令固定使用 `AGENTS.md`。
- 详细文档放入模块自己的 `docs/`。
- 文档文件名使用小写短横线，例如 `adding-a-tool.md`。
- 代码注释和项目文档默认使用中文。

## 15. 开发阶段

### 阶段 0：收敛旧扩展代码

- 审查当前未完成的 `src/extensions/` 和 `src/components/workspaceSidebar/`。
- 将可复用的注册表、测试或状态逻辑迁移到新结构。
- 删除已经迁移完成的旧入口，避免双注册表。
- 暂不保留 Workflow 一级分类。
- 迁移只改变目录和边界，不同时新增 Filerobot 业务。

### 阶段 1：独立拓展工作区

- 新增原应用顶层入口。
- 建立 `/app/extensions` 路径和独立页面外壳。
- 侧边栏只包含 Tools、Skills。
- 完成桌面和移动端基本导航。
- 原画廊和 Agent 页面行为保持不变。

### 阶段 2：Filerobot 图片编辑 Tool

- 建立 Tool 类型、注册表和懒加载宿主。
- 实现本地上传和现有图片库选择。
- 接入 Filerobot。
- 支持滤镜、裁剪、旋转、文字、水印和导出。
- 保存结果为新图片。
- 补齐 Tool 文档和测试。

### 阶段 3：Skills 管理

- 建立 Markdown Skill 类型、解析器和自动注册表。
- 展示平铺内置 Skill、来源和许可证。
- 完成 6 个经过当前 Agent 能力裁剪的开源 Skill。
- 暂不实现外部 Skill 上传。

### 阶段 4：手动 Skill 对话接入

- 在 Agent Composer 的 `@` 菜单显示 Skill。
- 使用包含稳定 ID 的结构化 atom，不根据可见文本猜测。
- 每条消息最多选择一个 Skill，不自动选择。
- 请求时注入当前 Skill 指令，消息正文保持干净。
- 持久化 Skill ID、名称和版本，编辑与重新生成沿用。

### 阶段 5：后续 Skill 管理

- 根据真实需求评估外部单文件导入。
- 外部文件按不可信输入校验格式、大小、ID 冲突和指令边界。
- 再决定是否支持编辑、导出、删除和版本保留。

### 阶段 6：后续能力

按真实需求逐项评估：

- AI 创建或修改 Skill。
- 更多独立 Tool。
- 多图拼接和自由画布。
- 背景移除。
- 视频处理 Tool。
- Tool 或 Skill 分享与 Remix。

这些能力不得提前污染第一阶段接口。

## 16. 测试策略

### 16.1 单元测试

- 路由解析。

- `Skills/builtins` 不直接导入文本模型和图片生成内部实现。
- `Tools` 与 `Skills` 不互相导入内部文件。
- 原业务代码不直接导入具体 Tool 或 `builtins/*.md`。

## 17. 每个模块的完成标准

一个 Tool 或 Skill 只有同时满足以下条件才算完成。Tool 需要独立目录和 README；Skill 需要单个完整 Markdown：

1. 拥有符合各自格式的完整说明和元数据。
2. 通过公共注册表接入。
3. 不绕过 adapters 访问原业务内部实现。
4. 用户流程完整，没有 stub、TODO 或不可用按钮。
5. 外部输入有与风险匹配的必要校验。
6. 失败路径有中文提示和必要日志。
7. 有对应单元测试或 UI 测试。
8. `npm run build` 和相关测试通过。
9. 行为、限制或目录变化已经同步到文档。

## 18. 当前非目标

- 不重做原有画廊和 Agent。
- 不根据对话内容自动选择 Skill。
- 不在一条消息中组合多个 Skill。
- 不让 Filerobot 成为全局图片处理引擎。
- 不开发通用 Fabric.js 编辑器。
- 不执行用户上传代码。
- 不建立 Tool/Skill 在线市场。
- 不开发视频剪辑。
- 不恢复 Workflow 一级入口。
- 不为未来能力提前引入复杂插件框架、事件总线或多 Store 同步。

## 19. 命名与迁移注意事项

- 目录名固定为 `src/Tools`、`src/Skills`、`src/ExtensionWorkspace`。
- React 组件使用 PascalCase 文件名。
- 普通 TypeScript 文件使用小写驼峰文件名。
- Tool、Skill ID 使用小写短横线。
- macOS 默认文件系统通常不区分大小写，迁移旧目录时不要依赖只修改大小写的普通重命名。
- 迁移当前未提交代码前先阅读并保留其有效实现，不使用破坏性 Git 命令。
- `src/extensions` 清理完成后，不保留指向新目录的长期兼容层；在同一迁移阶段修改调用方。

## 20. 最终原则

1. 原应用与拓展工作区 UI 独立。
2. Tools 与 Skills 概念独立。
3. 模块通过公共出口共享能力，Workspace Tool 继续通过 adapters 访问原业务数据。
4. 每个根模块先有说明和规范，再增加功能。
5. 每个具体 Tool 有自己的 README，每个内置 Skill 用一个 Markdown 完整自描述。
6. AI 开发前先读取对应模块文档，开发后同步更新文档。
7. Skills 保持单文件、手动选择和单消息单 Skill，不提前建设通用插件平台。
