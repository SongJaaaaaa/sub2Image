# 抠图模型自托管部署

## 1. 运行方式

一键抠图使用 `@imgly/background-removal`。图片不会上传到抠图服务器，浏览器只会下载模型和 WASM 运行资源，然后在用户设备中完成推理。

默认配置为空时，模型资源从 IMG.LY CDN 下载：

```text
https://staticimgly.com/@imgly/background-removal-data/1.7.0/dist/
```

生产环境可以把这些资源放到自己的服务器，再让浏览器从自己的域名下载。

## 2. 配置项

| 场景 | 配置项 | 示例 |
| --- | --- | --- |
| Vite 开发或直接构建 | `VITE_BACKGROUND_REMOVAL_PUBLIC_PATH` | `/models/background-removal/` |
| Docker 容器部署 | `BACKGROUND_REMOVAL_PUBLIC_PATH` | `/models/background-removal/` |
| 独立静态资源域名 | 对应上面任一配置 | `https://static.example.com/background-removal/` |

配置留空时继续使用 IMG.LY CDN。配置值可以是同源绝对路径或完整 HTTPS URL，末尾的 `/` 会自动补齐。

## 3. 下载模型资源

当前依赖版本为 `@imgly/background-removal@1.7.0`，模型资源版本必须保持一致。

工具默认使用完整精度 `isnet` 提升毛发和半透明边缘质量，用户也可以切换到 `isnet_fp16` 标准快速模式。下面下载的是完整 `dist` 目录，已经同时包含这两个模型；两种模式首次使用时会分别下载并缓存对应模型。

在项目根目录执行：

```bash
mkdir -p public/models/background-removal
mkdir -p /tmp/background-removal-data
curl -L https://staticimgly.com/@imgly/background-removal-data/1.7.0/package.tgz -o /tmp/background-removal-data/package.tgz
tar -xzf /tmp/background-removal-data/package.tgz -C /tmp/background-removal-data
cp -R /tmp/background-removal-data/package/dist/. public/models/background-removal/
```

最终目录必须是：

```text
public/models/background-removal/
├── 模型文件
├── onnxruntime-web/
└── 其他 background-removal-data 运行资源
```

不要把 `package/` 或 `package/dist/` 这两层目录复制到公开路径中。`VITE_BACKGROUND_REMOVAL_PUBLIC_PATH` 指向的目录应当直接包含 `dist` 内的文件。

模型文件体积较大，目录已经加入 `.gitignore`，不会被 Git 默认跟踪。构建和 Docker 打包前必须确保文件仍然存在于 `public/models/background-removal/`。

## 4. Vite 构建

在根目录 `.env` 中配置：

```env
VITE_BACKGROUND_REMOVAL_PUBLIC_PATH=/models/background-removal/
```

然后构建：

```bash
npm run build
```

Vite 会把 `public/models/background-removal/` 原样复制到：

```text
dist/models/background-removal/
```

部署整个 `dist/` 后，浏览器会从当前站点下载模型。

## 5. Docker 部署

先按第 3 节把模型下载到 `public/models/background-removal/`，再在部署 `.env` 中添加：

```env
BACKGROUND_REMOVAL_PUBLIC_PATH=/models/background-removal/
```

构建并启动：

```bash
docker compose --env-file .env -f deploy/compose.sub2api.yaml up -d --build
```

Docker 镜像会把模型文件复制到 Nginx 静态目录。容器启动时会把 `BACKGROUND_REMOVAL_PUBLIC_PATH` 注入前端构建产物，同一个镜像可以切换为空值或其他完整 URL。

## 6. 独立静态资源服务器

也可以把 `package/dist/` 上传到对象存储、CDN 或另一台静态服务器，然后配置完整地址：

```env
BACKGROUND_REMOVAL_PUBLIC_PATH=https://static.example.com/background-removal/
```

跨域托管时，静态服务器必须允许应用域名读取资源：

```http
Access-Control-Allow-Origin: https://image.example.com
```

建议同时配置：

```text
.wasm  -> Content-Type: application/wasm
.onnx  -> Content-Type: application/octet-stream
Cache-Control: public, max-age=2592000
```

同源 `/models/background-removal/` 不需要额外 CORS 配置。项目内置 Nginx 已为该目录配置 30 天缓存。

## 7. 验证

部署后打开浏览器开发者工具的 Network 面板，再执行一次抠图。

应当看到：

```text
GET /models/background-removal/...
```

或配置的静态资源完整 URL，并返回 `200`。不应再出现对 `staticimgly.com` 模型资源的请求。

如果返回 `404`，检查配置是否错误地指向了 `package/` 或 `package/dist/` 的上一级目录。如果出现 CORS 错误，检查静态资源服务器的 `Access-Control-Allow-Origin`。

## 8. 升级要求

升级 `@imgly/background-removal` 后必须同步：

1. 查看 `package-lock.json` 中的实际版本。
2. 下载相同版本的 `@imgly/background-removal-data`。
3. 替换服务器上的模型目录。
4. 更新本文档中的版本和下载 URL。
5. 运行 `npm run build`、`npm test` 和相关 E2E。
